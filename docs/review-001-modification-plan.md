### B. 主控台資料正確性
5. 主控台頁面的數值目前不是根據 Firestore 實際資料計算，需要改為真實統計。

## 二、整體修改策略

這次修改分成兩個層級：

1. 第一階段先完成低風險、立即可見的修正：品牌名稱、登出導向、登入頁視覺、側邊欄一致性、主控台真實數據。
2. 第二階段再做權限系統，因為它會影響所有頁面、資料規則與使用者流程，必須先設計清楚再實作。

## 三、建議開發階段

### Phase 1：品牌與基本體驗修正

目標：讓系統看起來像正式的大都會物業內部系統。

修改項目：
1. 全站文字 `RentalHub` 改為「大都會物業房務管理系統」。
2. 瀏覽器標題同步改名。
3. Header / Sidebar 品牌名稱同步改名。
4. 登出後導向 `index.html`。
5. 側邊欄項目統一 class、icon 寬度、文字大小、padding、高度。
6. 登入頁參考舊站風格重做：背景、公司品牌、登入卡片、返回首頁連結。

驗收標準：
- 全站不再出現 `RentalHub` 顯示字樣。
- 登出後回首頁。
- 每個後台頁側邊欄排版一致。
- login 頁視覺與大都會物業品牌一致。

預估風險：低。

---

### Phase 2：主控台真實數據

目標：主控台不再顯示假數字或靜態數字。

主控台應從 Firestore 實時計算：
1. 建案數量：`/users/{uid}/projects`。
2. 房間總數：所有 `projects/{projectId}/rooms` 加總。
3. 已出租房數：room.status = `已出租` 或有 active contract。
4. 空置房數：room.status = `空置`。
5. 房客數：`/users/{uid}/tenants`。
6. 本月應收租金：本月 rentRecords 或 active contracts 加總。
7. 本月已收租金：本月 rentRecords.paid 加總。
8. 逾期租金：本月 amount - paid 且 status 非已繳。
9. 待處理修繕：maintenanceTasks status = `待處理` / `進行中`。
10. 即將到期合約：contracts.end 距今天 60 天內。

驗收標準：
- 新增建案/客房/房客/合約後，主控台數字會變動。
- 無資料時顯示 0 與空狀態，不顯示假資料。
- Console 無錯誤。

預估風險：中低。

---

### Phase 3：帳號管理資料模型

目標：登入後自動建立使用者帳號資料，提供後續權限系統使用。

建議資料路徑：

```text
/users/{uid}/profile/main
```

或若要支援同一公司多使用者，建議後續升級為：

```text
/organizations/{orgId}/members/{uid}
```

短期純前端 MVP 建議先用：

```text
/users/{uid}/account/profile
```

欄位：

```js
{
  uid,
  displayName,
  email,
  photoURL,
  role: '管理員',
  status: '啟用',
  createdAt,
  updatedAt,
  lastLoginAt
}
```

第一個登入者預設為「管理員」。
後續新增帳號時，由管理員調整角色。

驗收標準：
- Google 登入後自動建立/更新帳號資料。
- 帳號管理頁能看到登入者姓名、Email、頭像、角色、狀態。
- 管理員可調整角色。

預估風險：中。

---

### Phase 4：角色權限規則

目標：依角色控制側邊欄、頁面可見性、操作按鈕與資料寫入。

角色設計：

#### 管理員
- 可瀏覽與操作所有頁面。
- 可管理帳號與角色。

#### 員工
- 可瀏覽所有後台頁面。
- 可新增修繕/清潔工單。
- 不可刪除核心資料，例如建案、房客、合約、帳號。

#### 房務
- 只能看到派工/清潔相關頁面。
- 只能操作派發給自己的工單。

#### 工務
- 只能看到修繕派工頁面。
- 只能操作派發給自己的修繕工單。

#### 租客
- 只能看到自己的租屋資訊。
- 可填寫水電瓦斯度數。
- 可回報修繕或清潔請求。
- 可查看繳費/租金/帳單狀態。

#### 訪客
- 只能看到建案管理內的未出租資訊。
- 不可看到房客、合約、租金、內部資料。

實作層級：
1. 前端：依角色隱藏側邊欄與按鈕。
2. 前端：進頁前檢查權限，不符則導回首頁或顯示無權限。
3. Firestore rules：後續需要同步加強，避免只靠前端隱藏。

驗收標準：
- 不同角色登入後看到不同選單。
- 無權限者不能直接輸入 URL 進入不該看的頁面。
- 無權限操作按鈕不顯示。

預估風險：中高，因為會牽涉所有頁面。

---

### Phase 5：租客與訪客專用頁面

目標：讓租客/訪客不是看到管理後台，而是看到簡化入口。

建議新增頁面：
1. `tenant-portal.html`：租客入口。
2. `guest-listings.html`：訪客可看未出租房源。
3. `accounts.html`：帳號管理。

租客入口功能：
- 我的租屋資訊。
- 水電瓦斯度數填寫。
- 修繕/清潔請求。
- 租金與帳單狀態。

訪客入口功能：
- 只顯示空置房間。
- 不顯示房東內部資料、租金收款、房客資料。

驗收標準：
- 租客角色登入後進 `tenant-portal.html`。
- 訪客角色登入後只看到空置資訊。
- 管理後台資料不外洩。

預估風險：中高。

## 五、開發與 QA 方式

每一階段完成後都要跑：

```bash
python3 -m json.tool firebase.json >/dev/null
python3 -m json.tool firestore.indexes.json >/dev/null
for f in public/js/*.js; do node --check "$f"; done
```

並做瀏覽器煙霧測試：
1. 未登入不可進後台。
2. 登入頁 console 無錯誤。
3. 登出導向正確。
4. 修改頁面 console 無錯誤。
5. 正式站部署後確認 live HTML/JS 已更新。
